VignOCR trains RF-DETR on the Narval cluster (Digital Research Alliance). Allocation: account `def-khenni`, user `ydait`, project `6075472`. Real project path `/lustre06/project/6075472/ydait/pharmakon-ocr` (symlinked `~/projects/def-khenni/ydait/pharmakon-ocr`). venv lives on scratch `/lustre07/scratch/ydait/vignocr/venv`. Modules: StdEnv/2023, python/3.11, cuda/12.6. Compute nodes are air-gapped (no internet).

**The recurring failure pattern (as of 2026-06-05):** validation jobs pass, every training job (02 detection / 02a vignette) has failed — always a Python dependency/provisioning defect on the offline node, never GPU/CUDA/SLURM/architecture. Fixed reactively one ImportError at a time: RC-1 VIGNOCR_ACCOUNT not propagated → RC-2 EXIT-trap unbound `jlog` cascade-cancelled the afterok DAG → RC-3 `sympy` missing (torch._dynamo lazy dep) → RC-4 pretrained weights uncached → **RC-5 (live): `pytorch_lightning` missing because pyproject pins bare `rfdetr>=1.1` not `rfdetr[train,loggers]`** — rfdetr lazy-imports its Lightning training stack inside `model.train()`, so it crashes mid-training, and the setup verifier only checks the inference import chain (`rfdetr`), never `rfdetr.training`.

**Why:** offline wheelhouse + from-PyPI deps provisioned reactively. **How to apply:** declare `rfdetr[train,loggers]`, verify the *training* chain (`import rfdetr.training`), smoke-test one epoch in `salloc` before submitting the afterok DAG (one non-zero exit cancels all downstream stages). Also fix `train.py` kwargs: `lr_backbone`→`lr_encoder`, drop `clip_max_norm`/`num_workers` (not in rfdetr's `train()` API). See [[vignocr-dataset-gaps]].

**Install gotchas (2026-06-12, v2 round):** (1) ALWAYS activate the venv before pip — `module load python/3.11 && source ~/scratch/vignocr/venv/bin/activate`. Running pip from a bare login shell uses the **system pip 21.x on Python 3.9** (`Defaulting to user installation`), whose old `packaging` throws `TypeError: expected string or bytes-like object` resolving any package-with-extras (uvicorn[standard], rfdetr[train,loggers], python-doctr[torch]). The venv's upgraded pip is immune. (2) The canonical installer is `bash scripts/setup_narval.sh` (it builds+activates the venv and now installs BOTH `[ml]` and the v2 deps); do NOT hand-run `pip install --no-index -e .[ml,v2]` — the heavy v2 extra resolution is fragile and the wheelhouse may lack transformers/python-doctr. (3) TORCH-CLOBBER: never `pip install python-doctr[torch]` ONLINE — it can replace the Compute Canada CUDA torch with a generic PyPI CPU wheel and silently break GPU. Install v2 deps wheelhouse-first (`--no-index`, torch-safe); online fallback uses `--no-deps` for doctr. setup_narval.sh + fetch_pretrained_v2.sh now guard this (compare `torch.version.cuda` before/after, vdie on loss). (4) v2 jobs (12 build-dataset / 13 train-donut / 14 compare) need transformers+sentencepiece (Donut) and doctr (v2b only now) in the SCRATCH venv before submit — `submit_v2.sh` preflights this on the login node.

**docTR/h5py/opencv soname race (2026-06-21):** `import doctr` → `doctr.io` imports cv2 (the `opencv` module) BEFORE `doctr.models` imports h5py. The opencv module links an OLDER libhdf5 with the SAME soname (`libhdf5.so.310`), so it wins the load race and h5py's `libhdf5_hl` can't resolve `H5T_IEEE_F16BE_g` (float16, added HDF5 1.14.4) → `ImportError: ... undefined symbol`. Loading `module load hdf5/1.14.6` is NOT enough (cv2's lib already loaded via its own path). **Fix in lib.sh:** after loading hdf5, `export LD_PRELOAD=$EBROOTHDF5/lib/libhdf5.so[:libhdf5_hl.so]` so the newer libhdf5 wins process-wide (set VIGNOCR_MODULE_HDF5=hdf5/1.14.6). **More important fix:** the VLM-dataset value backend (scripts/build_vlm_dataset.py) was changed to DEFAULT to **TrOCR** (`microsoft/trocr-base-printed`, pure transformers — Narval-clean, AutoProcessor for 4.x/5.x), so the entire v2a path (12→13→14) NO LONGER depends on docTR. docTR is now needed ONLY for v2b (full-page); if it still won't import, v2b is skipped by compare.py and v2a + v1 are unaffected. Wheelhouse confirmed present: transformers 5.10.1, h5py 3.16.0, python-doctr 1.0.1, torch 2.12.0 — all `+computecanada`. NOTE: transformers **5.10.1** is much newer than the code was written against (~4.x); Donut under 5.x — now uses `AutoProcessor` (DonutProcessor's import path moved in 5.x) but the full train path is still UNVERIFIED; if stage 13 fails, that's the suspect.

**META-ROOT-CAUSE — stale Narval checkout (2026-06-22):** the single most recurring failure is NOT a code bug — it's that Narval runs an OLD `git` checkout because a `git pull` was skipped. Job 63684084 failed on the h5py/doctr default even though the TrOCR fix was already on origin: Narval was at `21bb809-dirty`, two commits behind. Origin is `github.com/qmrx87/pharmakon-ocr` (branch `master`); desktop HEAD == origin and IS the source of truth. **Fix shipped (commit 860a6e6):** `slurm/lib.sh: vignocr_warn_if_stale()` runs `git fetch` at the top of `submit_all.sh`/`submit_v2.sh` and `vdie`s if the checkout is BEHIND upstream (override `VIGNOCR_SKIP_GIT_CHECK=1`); also warns on a dirty tree. The submit_v2 preflight now does REAL imports (not `find_spec`, which falsely passed broken `doctr`). **Narval recovery when dirty+behind:** `cd <repo> && git fetch && git reset --hard origin/master` (Narval's local dirty edits are superseded by origin), then re-run `scripts/fetch_pretrained_v2.sh` (it now caches TrOCR too) before `slurm/submit_v2.sh`. **How to apply:** I edit on the Windows desktop → commit+push to origin/master → user MUST `git pull` (or reset --hard origin/master) on Narval before running. The guard now enforces this.
